# 第 18 课：Tiny GPT 训练强化

## 这一课为什么不再改模型结构？

前几课已经把 Tiny GPT 的核心结构拼出来了：

- token embedding
- position embedding
- multi-head causal self-attention
- residual connection
- layer norm
- feedforward MLP
- lm head

这一课的重点不再是“模型里多加一层什么模块”，而是把训练过程做得更完整、更可观察、更可恢复。

真实训练工程里，下面这些事情和模型结构同样重要：

- 训练集和验证集分开看
- 定期记录 train / val loss
- 保存 checkpoint
- 中途生成样本观察质量
- 保存超参数和日志
- 训练结束后能复现当时的实验条件

## 一个更完整的 Tiny GPT 训练脚本应包含什么？

一个最小但实用的训练脚本，通常至少要有：

- 数据准备
- tokenizer / vocab / encode / decode
- train / val split
- 模型定义
- loss 函数
- optimizer
- 训练循环
- 定期评估
- checkpoint 保存
- 生成样本
- 日志和配置保存

如果少了这些部分，训练就很难复盘，也不容易比较不同实验。

## train / val split 的意义

训练集 `train` 用来更新参数。

验证集 `val` 不参与参数更新，只用来观察模型在“没直接拿来训练的数据”上的表现。

这样做的意义是：

- `train loss` 反映模型对见过数据的拟合程度
- `val loss` 反映模型的泛化情况

如果只看训练集，很容易误以为模型越来越好，但实际上它可能只是在记忆训练文本。

## loss 曲线怎么看？

通常会同时看两条曲线：

- train loss
- val loss

常见情况：

- 两条都下降：模型确实在学
- train loss 下降，val loss 也下降但更慢：正常
- train loss 持续下降，val loss 停住甚至回升：开始过拟合
- 两条都不怎么动：学习率可能不合适，或者模型 / 数据设置有问题

所以 loss 曲线不是只看“有没有下降”，而是看 train 和 val 的关系。

## checkpoint 的作用

checkpoint 就是训练中途保存下来的状态。

至少应该保存：

- 模型参数

更完整一些还可以保存：

- optimizer state
- 当前 step
- 当时的 train / val loss

它的作用包括：

- 训练中断后可以继续
- 可以回到表现更好的某个时刻
- 可以拿某个中间模型直接做生成测试

这一课里，checkpoint 至少包含模型权重，并额外保存了 optimizer state 和元信息。

## 生成样本为什么重要？

只看 loss 不够。

因为语言模型的目标虽然是 next-token loss，但我们最后真正关心的是：

- 生成文本是否更像训练文本
- 结构是否更稳定
- 重复是否减少
- 是否开始出现像样的短句

定期保存生成样本，可以让我们看到：

- loss 下降时，文本质量是否也在变化
- 某一阶段是不是已经开始记忆训练语料
- 不同 checkpoint 的文本风格差别

## 超参数记录为什么重要？

如果一次实验训练得不错，但你没有记录超参数，后面几乎无法严格复现。

至少应该记录：

- block_size
- batch_size
- n_embd
- num_heads
- num_layers
- learning_rate
- max_iters
- eval_interval
- vocab_size
- 数据长度

这样以后才能回答：

- 这次效果为什么比上次好？
- 是模型变大了，还是学习率更合适？
- 是训练更久了，还是数据更多了？

## 什么是过拟合？

过拟合就是模型对训练数据学得太死，导致：

- 在训练集上 loss 很低
- 在验证集上提升不明显，甚至变差

对 Tiny GPT 来说，尤其是在很小的字符级语料上，过拟合非常常见。

因为模型参数不少，但数据很少，它很容易把常见句子模式直接记下来。

## 为什么小语料上的 Tiny GPT 容易记忆训练文本？

原因很直接：

- 数据量小
- 文本模式重复多
- 字符级建模更容易背局部片段
- 模型容量已经足够覆盖这份小语料

所以训练一段时间后，模型常常会出现：

- 生成内容越来越像训练文本原句
- 某些短句被反复复述
- 结构是对的，但新意不多

这不是模型坏了，而是数据规模和任务设定决定的。

## 下一步如何过渡到数据管线和工程化重构？

这一课把训练过程补齐后，下一步就可以更自然地进入：

- 更清晰的数据管线
- dataset / batch 逻辑拆分
- 配置管理
- 训练和推理脚本分离
- 更规范的 checkpoint 管理
- 更大的语料和更长的 context

也就是说，后面要做的不是“随便再堆更多层”，而是把训练代码逐步变成更稳定、更可维护的工程结构。
